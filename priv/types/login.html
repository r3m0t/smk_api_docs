An example login payload for first establisting a session is described in the seq type above.

To resume a session the field for this in the login payload needs to be specified with the identifier originally returned in the incoming login-response within the message with seq 1. The message wrapping the login payload also needs to have the outgoing seq that your application was previously going to send next before the connection was dropped. Your incoming sequence number also needs to be as it would have been before the connection was dropped so that any messages sent to your application from Smarkets while the connection was down can be replayed. This is done by comparing the sequence of the resulting login-response and the one your application is expecting to be next. Smarkets will do the same by comparing the sequence of your login payload with what it expects to be the next message and may also send a replayed in your application's direction.
