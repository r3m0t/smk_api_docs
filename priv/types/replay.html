When the incoming message's sequence is greater than the one the server or your application is expecting then a replay payload needs to be sent to fill in the gap and bring that side in line with the other. An example of this can be found in the description of login. The seq for replay should not be confused with the seq for the message containing this payload. The seq here needs to be the sequence that is being requested for playback.

When receiving a message with a replay payload every message from the sequence inside the replay payload upwards should be resent to the other side of the connection with their message sequence entact and replay set to true. Both the server and clients need to do this.

A number of message payloads should not be re-sent and instead replaced with gapfill:

  • login
  • login-response
  • replay

An example of this can be seen when resuming a session for server outgoing sequence 58 which was originally a login-response.
