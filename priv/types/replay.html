When the incoming sequenced's seq is greater than the one the server or your application is expecting then a replay message needs to be sent to fill in the gap and bring that side in line with the other. An example of this can be found in the description of login. The seq for replay should not be confused with the seq for the sequenced containing this message. The seq here needs to be the sequence that is being requested for playback.

When receiving a sequenced payload with a replay message every message from the seq inside the replay message upwards should be resent to the other side of the connection with their sequenced seq entact and replay set to true. Both the server and clients need to do this.

Some messages should not be re-sent and instead replaced with gapfill:

  • login
  • login-response
  • replay

An example of this can be seen when resuming a session for server outgoing sequence 58 which was originally a login-response.
